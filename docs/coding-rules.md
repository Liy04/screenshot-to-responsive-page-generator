# 编码规范

本文档记录项目后续开发的基础编码规则。当前阶段只编写工程骨架、占位页面、健康检查和 smoke 脚本，不编写复杂业务代码。

## 通用规则

- 每次任务先计划、再执行、最后验证。
- 保持改动小而清晰。
- 文件命名要表达用途。
- 不提交依赖目录、构建产物、临时文件。
- 不在代码中写真实账号、密码、Token 或数据库凭据。
- 注释用于解释复杂意图，不写重复代码含义的空注释。

## 前端规则

技术栈：Vue3、Vite、HTML、CSS、JavaScript。

- 使用 Vue3 的组合式 API（Composition API，按功能组织逻辑）。
- 页面组件放在清晰的功能目录中。
- API 请求统一封装，不在组件里散落请求地址。
- 样式优先保持简单，先保证可读性和响应式。
- 组件命名使用清晰的业务名。
- 避免把复杂业务逻辑直接写进模板。

## 后端规则

技术栈：Java、Spring Boot、MyBatis-Plus、RESTful API。

- Controller（控制器）负责接收请求和返回响应。
- Service（服务层）负责业务逻辑。
- Mapper（数据访问层）负责数据库访问。
- 实体、DTO（数据传输对象）、VO（视图对象）命名要清晰区分。
- API 路径遵循 RESTful 风格，围绕资源设计。
- 异常处理和响应结构保持统一。

## Worker 规则

技术栈：Python。项目推荐目标版本为 Python 3.11；Week 01 smoke 脚本允许兼容 Python 3.10+。

- Worker 只负责独立处理任务，不直接承担 Web 接口职责。
- Week 01 优先使用 Python 标准库，不为 smoke 脚本引入额外依赖。
- 输入和输出结构要清晰，方便后端调用。
- 生成逻辑要分步骤组织，避免一个文件过大。
- 重要处理步骤需要保留日志。
- 出错时返回明确错误信息，便于后端更新任务状态。

## 文档规则

- 文档使用中文。
- 技术名词可以保留英文，但首次出现时尽量补充中文解释。
- 文档要说明“当前做什么”和“当前不做什么”。
- 重要设计变更要同步更新文档。
