# Week 01 计划

## 文件目的

本文件用于明确第一周的目标、范围、每日任务、交付物、验收标准与风险控制方式。

它既是“本周任务说明书”，也是后续让 Codex 读取的阶段性上下文文档。

## 本周主题

第一周主题：**初始化，不求功能完整，只求边界清楚、骨架成立、规则稳定。**

## 本周目标

第一周只做三件大事：

1. 确定 MVP 边界
2. 建立工程骨架
3. 固化 Codex 协作规则

## 本周完成定义

如果一周结束时满足以下条件，就算完成：

- 仓库文档齐全，可指导后续开发
- 前端 / 后端 / worker 至少能各自独立启动
- Codex 进入仓库后能根据文档正确理解当前阶段
- 已形成最小 smoke 测试路径
- 第二周可以直接进入“最小闭环任务开发”

## 本周范围

本周允许做的内容：

- 文档编写
- 目录初始化
- 前端工程初始化
- 后端工程初始化
- worker 脚本初始化
- 健康检查接口
- 占位首页
- smoke 测试说明

## 本周不做

本周明确不做：

- 真实截图识别
- 真实 Figma 数据接入
- 真实页面生成
- 真实 AI 调用
- Redis / RabbitMQ 接入
- 多页面任务编排
- 复杂回归测试体系
- 页面可视化编辑器

## 本周建议工作方式

### 人负责什么
- 负责决定方向
- 负责控制范围
- 负责审查计划
- 负责验收结果

### Codex 负责什么
- 阅读仓库上下文
- 输出计划
- 修改代码或文档
- 运行验证
- 汇报变更结果

## 每日协作节奏

每天建议按这个节奏走：

1. 先阅读本文件，确认今日主题
2. 给 Codex 下发“单一主题任务”
3. 先审查计划
4. 再执行改动
5. 最后检查验证结果
6. 把当日结论写回文档或提交记录

## 风险与应对

| 风险 | 表现 | 应对方式 |
|---|---|---|
| 任务范围失控 | 刚做初始化就想接模型、接 Figma、接队列 | 以本文件“本周不做”为强约束 |
| 文档与代码不一致 | README 写的命令和真实命令不同 | Day 5 统一做一致性检查 |
| Codex 改动过大 | 一次任务改很多无关文件 | 每次任务只允许一个明确主题 |
| 启动链路依赖过重 | 一上来就依赖 MySQL / Redis / MQ | 第一周先保证不依赖这些也能 smoke |
| 会话上下文漂移 | 聊着聊着偏离当日目标 | 新任务尽量新开窗口，并重读文档 |

## 关键字段说明

| 字段 | 作用 | 更新时机 |
|---|---|---|
| 本周主题 | 统一本周心智模型 | 本周开始时确定 |
| 本周目标 | 限定本周的真正目标 | 周中发生重大调整时更新 |
| 本周不做 | 防止范围膨胀 | 若阶段策略变更时更新 |
| 每日任务表 | 作为每日执行入口 | 每日复盘后可微调 |
| 风险与应对 | 帮助快速识别偏航 | 有新风险时补充 |
